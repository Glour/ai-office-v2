# Систематическая отладка - Debugging без гаданий

## 📦 Установка

1. Скопируйте папку `skills/systematic-debugging/` в ваш workspace OpenClaw
2. OpenClaw автоматически подхватит скилл при следующем сообщении
3. Проверка: скажите агенту "У меня баг/ошибка/не работает X"

## 🎯 Что умеет

Превращает хаотичную отладку методом тыка в системный процесс:
- Находит реальную причину, а не симптом
- Экономит время - один точный фикс вместо 10 попыток
- Предотвращает новые баги от рандомных правок
- Работает для любых технических проблем

## ⚙️ Как работает

## ⚡ Железное правило

```
НИКАКИХ ФИКСОВ БЕЗ ПОНИМАНИЯ ПРИЧИНЫ
```

Рандомные фиксы тратят время и создают новые баги. Если не прошёл Фазу 1 - нельзя предлагать решения.

## Когда использовать

При любой технической проблеме:
- Скрипт падает или выдаёт ошибку
- Сервис не запускается
- Код работал, теперь нет
- Неожиданное поведение
- "Странно, должно было сработать"

**Особенно** когда:
- Под давлением времени
- "Очевидный фикс" напрашивается
- Уже пробовал 2+ решения без успеха

## 🔄 Четыре фазы отладки

### Фаза 1: Расследование причины

**ПЕРЕД любым фиксом:**

**1. Прочитай ошибку целиком**
   - Не пропускай стектрейсы и warnings
   - Часто ответ прямо в тексте ошибки
   - Запиши: файл, строка, код ошибки, точное сообщение

**2. Воспроизведи проблему**
   - Можешь повторить её стабильно?
   - Какие точные шаги нужны?
   - Если не воспроизводится - собирай больше данных, не гадай

**3. Что изменилось?**
   - `git log`, последние коммиты
   - Обновления системы, библиотек, OpenClaw?
   - Новые зависимости, конфиги?
   - Что было сделано перед проблемой?

**4. Проследи поток данных**
   Для многокомпонентных систем (пользователь → API → база → ответ):
   ```
   На каждом стыке:
   - Что входит?
   - Что выходит?
   - Где именно ломается?
   ```
   Используй логи, print/console.log, breakpoints.

### Фаза 2: Анализ паттерна

**1. Найди рабочий пример**
   - Есть похожий код/скрипт/конфиг который работает?
   - В документации, примерах, других проектах?

**2. Сравни**
   - Что отличается между рабочим и сломанным?
   - Версии? Зависимости? Конфигурация?
   - Используй diff инструменты

**3. Проверь зависимости**
   - Всё ли установлено (пакеты, библиотеки, сервисы)?
   - Правильные версии?
   - Доступны пути, порты, права доступа?

### Фаза 3: Гипотеза и тест

**1. Сформулируй гипотезу**
   - "Я думаю причина в X потому что Y"
   - Конкретная и проверяемая

**2. Минимальный тест**
   - ОДНО изменение за раз
   - ОДНА переменная
   - Контролируемые условия

**3. Проверь результат**
   - Сработало → Фаза 4
   - Не сработало → новая гипотеза, вернись к пункту 1

**4. НЕ лепи фиксы поверх**
   - Если не сработало, ОТКАТИ изменение
   - Не накапливай непонятные правки
   - Каждая итерация - чистый тест

### Фаза 4: Реализация фикса

**1. Фикси причину, не симптом**
   - Симптом: "добавлю sleep чтобы не падало"
   - Причина: "исправлю race condition"

**2. Одно изменение за раз**
   - Не смешивай несколько фиксов
   - Каждое изменение должно быть обосновано

**3. Проверь полностью**
   - Проблема ушла?
   - Ничего другого не сломалось?
   - Проверь граничные случаи

**4. Запиши решение**
   - Что было, что изменил, почему сработало
   - Поможет в будущем

**5. Правило трёх попыток**
   - Если 3+ фикса не сработали → СТОП
   - Проблема системная/архитектурная
   - Нужен другой подход или консультация

## 🚩 Красные флаги - ОСТАНОВИСЬ

Если ловишь себя на мысли:

- ❌ "Быстрый фикс, потом разберусь"
- ❌ "Попробую поменять X, вдруг поможет"
- ❌ "Добавлю несколько изменений сразу"
- ❌ "Не совсем понимаю, но может сработает"
- ❌ "Ещё один фикс..." (когда уже 2+ не сработали)

→ **СТОП. Вернись к Фазе 1.**

## 💡 Примеры использования

**Пример 1:** "Скрипт падает с ошибкой 'module not found'"
→ Агент пройдёт все фазы: проверит импорты, виртуальное окружение, версии пакетов, найдёт реальную причину.

**Пример 2:** "Docker контейнер не запускается"
→ Системно проверит логи, конфиг, зависимости, сеть, права - найдёт точную проблему.

**Пример 3:** "Код работал вчера, сегодня нет"
→ Найдёт что изменилось (git, обновления, конфиги), воспроизведёт, выяснит причину.

## 🔧 Настройка под себя

### Типичные проблемы вашей системы

Создайте таблицу частых проблем и где искать:

```markdown
| Проблема | Первым делом проверь |
|----------|---------------------|
| API не отвечает | Логи, порт, firewall |
| База не подключается | Креденшалы, сетевой доступ, сервис работает |
| Импорт не работает | venv активен, пакет установлен, версия |
| Скрипт не запускается | Права (chmod +x), shebang, путь |
```

### Инструменты отладки

Настройте удобные команды:

```bash
# Логи сервиса
alias service-logs="docker logs -f myservice"

# Проверка портов
alias check-ports="lsof -i -P -n | grep LISTEN"

# Git изменения
alias recent-changes="git log --oneline -10"

# Статус зависимостей
alias deps-check="npm list --depth=0"  # для Node.js
alias deps-check="pip list"            # для Python
```

### Логирование

Добавьте детальное логирование в критичные места:

```python
# Python
import logging
logging.basicConfig(level=logging.DEBUG)
logger = logging.getLogger(__name__)
logger.debug(f"Переменная X = {x}")
```

```javascript
// JavaScript
console.log('[DEBUG] Состояние:', state);
```

### Документирование решений

Создайте файл `debugging-log.md` с историей проблем и решений:

```markdown
## 2026-03-06 - Скрипт падал с Permission Denied

**Проблема:** backup.sh падал при запуске  
**Причина:** Забыл chmod +x после редактирования  
**Решение:** `chmod +x backup.sh`  
**Урок:** Проверять права после редактирования скриптов
```

## 🎓 Философия

**Понимание > Скорость**

Потратить 10 минут на понимание причины лучше чем час на рандомные фиксы.

**Один точный выстрел > Очередь**

Системная отладка находит корень проблемы с первого раза.

**Знание накапливается**

Каждая отладка - урок. Документируй и не повторяй ошибки.
