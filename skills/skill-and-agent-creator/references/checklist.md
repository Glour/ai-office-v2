# Чеклист создания агентов и скиллов

> Синхронизирован с `SKILL.md` (обновлено 09.03.2026)
> Источник: боевой опыт с 39 скиллами и 6 агентами

---

## 🤖 СОЗДАНИЕ АГЕНТА (10 шагов)

### Фаза 1: Планирование

- [ ] Определить роль/задачу агента
- [ ] Выбрать тип: **полноценный** (свой бот, память, скиллы) / **специализированный** (своя экосистема) / **маска** (systemPrompt в топике)
- [ ] Определить tools: что нужно, что запретить
- [ ] Определить memorySearch: true для рабочих агентов
- [ ] Определить связи с другими агентами

### Фаза 2: Конфиг

- [ ] Добавить агента в `agents.list[]` в openclaw.json
- [ ] Обязательные поля: `id`, `name`, `model`, `workspace`, `agentDir`
- [ ] Модель: ПОЛНОЕ имя (`anthropic/claude-sonnet-4-6`), НЕ алиас!
- [ ] Workspace: `~/.openclaw/agents/<id>/agent/`
- [ ] `memorySearch.enabled`: true для рабочих, false для масок/изолированных
- [ ] `heartbeat.every`: "0" если мониторинг не нужен
- [ ] `tools.deny`:
  - `gateway` - ВСЕГДА запрещать
  - `cron` - если не нужны кроны
  - `exec` - если чисто текстовый агент

### Фаза 3: Связи и Binding

- [ ] Добавить id в `agentToAgent.allow[]`
- [ ] Binding в конфиге: `accountId` или `topicId`
- [ ] Если Telegram бот: `accounts.<id>.botToken`
- [ ] Если топик: `groups.<gid>.topics.<tid>.requireMention: false`
- [ ] `sessions_send` всегда с `timeoutSeconds=0`

### Фаза 4: Workspace (12 файлов)

- [ ] `mkdir -p ~/.openclaw/agents/<id>/agent/memory`
- [ ] **AGENTS.md** - роль, прозрачность, команда, скиллы, память, автоулучшение
- [ ] **SOUL.md** - личность, принципы, границы
- [ ] **USER.md** - профиль владельца адаптированный под роль агента
- [ ] **IDENTITY.md** - имя, роль, краткое описание
- [ ] **MEMORY.md** - сводка ключевых фактов
- [ ] **TOOLS.md** - реальные инструменты с командами
- [ ] **memory/lessons.md** - уроки и правила
- [ ] **memory/patterns.md** - паттерны автоулучшения
- [ ] **memory/projects-log.md** - история задач
- [ ] **memory/architecture.md** - самоописание агента (рекомендуется)
- [ ] Симлинк на скиллы: `ln -s ~/path/to/skills skills`

Шаблоны всех файлов: `references/agent-templates.md`

### Фаза 5: Гигиена

- [ ] Ночная чистка покрывает `~/.openclaw/agents/*/sessions/` (wildcard)
- [ ] SQLite общая база чистится существующим кроном

### Фаза 6: Перезапуск

- [ ] `cp ~/.openclaw/openclaw.json ~/.openclaw/openclaw.json.bak`
- [ ] Перезапуск через `gateway config.patch` или `launchctl kickstart`
- [ ] ⚠️ НЕ `openclaw gateway restart` из сессии агента!
- [ ] Подождать 10-15 сек → `openclaw status`

### Фаза 7: Тест

- [ ] Отправить сообщение → получить ответ
- [ ] tools.deny работает (агент не может рестартить gateway)
- [ ] memorySearch изолирован (не видит чужое если отключён)
- [ ] sessions_send доходит (agentToAgent связь)
- [ ] Пишет уведомления прозрачности (🔄/📚/🔍)
- [ ] При старте читает memory/lessons.md, patterns.md, projects-log.md
- [ ] Перед задачей читает SKILL.md (если есть скиллы)

### Фаза 8: Выравнивание с командой

- [ ] Все файлы из Фазы 4 на месте
- [ ] USER.md заполнен под его роль (не пустой шаблон!)
- [ ] Маршрутизация по скиллам в AGENTS.md
- [ ] Знает про команду (таблица агентов)
- [ ] Другие агенты знают про нового (обновить их AGENTS.md)

### Фаза 9: Командная архитектура (для рабочих агентов в команде!)

- [ ] **Конституция** подключена в TOOLS.md (`references/team-constitution.md`)
- [ ] **Board-First Protocol** прописан в TOOLS.md (board = основной канал, sessions_send = будильник)
- [ ] **BOOTSTRAP.md** создан (шаблон в `references/agent-templates.md`)
- [ ] **Briefing pattern** описан (projects/НАЗВАНИЕ/briefing.md для каждой задачи)
- [ ] **ОТК цепочка** определена (кто проверяет результат перед доставкой)
- [ ] **Plan + "Погнали"** — координатор шлёт план, владелец подтверждает
- [ ] **Defrost protocol** — что делать если агент завис >3 мин
- [ ] **Smart trash** — скрипт сброса сессии покрывает нового агента
- [ ] **agentToAgent allow** — добавлен в список если нужен синхронный вызов
- [ ] **active-projects.md** — обновлён (если есть)

## 📝 СОЗДАНИЕ СКИЛЛА (11 шагов)

### Фаза 1: Планирование

- [ ] Определить тип: workflow / role / data-driven / гибрид
- [ ] Определить триггерные фразы
- [ ] Спланировать структуру (SKILL.md + references/ + data/ + scripts/)

### Фаза 2: SKILL.md

- [ ] YAML frontmatter: `name`, `description` с триггерами
- [ ] ⚠️ Поле `version` НЕ поддерживается!
- [ ] Алгоритм / роль / работа с данными (по типу)
- [ ] 2-3 реалистичных примера
- [ ] Данные в `data/`, НЕ в `memory/`!
- [ ] Размер: 100-300 строк (мета-скиллы до 500)

### Фаза 3: Проверка

- [ ] Self-contained (работает без внешних знаний)
- [ ] Нет TODO/заглушек
- [ ] Триггеры покрывают варианты обращения

### Фаза 4: Безопасность (публичные)

- [ ] `grep -ri "личные_данные\|пути\|токены"` = 0 результатов
- [ ] Model-agnostic (работает на разных моделях)

### Фаза 5: Тест и итерация

- [ ] Реальный запрос → проверить результат
- [ ] Edge cases (пустой ввод, нестандартный запрос)
- [ ] {{OWNER_NAME}} поправил → записать, 3+ повтора → добавить в скилл

## ⛔ Когда НЕ создавать

**Не создавай скилл если:**
- Задача одноразовая (просто напиши промпт)
- Нет повторяемости (делаешь это раз в год)
- Модель и так знает (не добавляешь уникального)

**Не создавай агента если:**
- Хватит маски (systemPrompt в топике)
- Задача не требует своей памяти и состояния
- Нет кому им пользоваться регулярно

**Миграция маска → агент:**
Если маска начала: накапливать контекст между сессиями, нуждаться в своих файлах данных, требовать отдельных tools - пора переводить в полноценного агента (все 10 шагов).

## 📋 Экспресс-чеклист (copy-paste)

### Полноценный агент:
```
1. Тип → конфиг → binding → workspace (12 файлов) → гигиена → рестарт → тест → выравнивание → командная архитектура
```

### Скилл:
```
1. mkdir skills/<имя>/ → SKILL.md (frontmatter + алгоритм + примеры) → тест → если публичный: grep + public версия
```
