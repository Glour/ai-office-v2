# Техническое задание
# Персональная агентная команда Алекса

Дата: 2026-04-09
Статус: рабочее ТЗ перед проектированием и разработкой

## 1. Цель

Построить персональную многоагентную систему для Алекса, в которой:

- есть один главный Orchestrator;
- есть несколько специализированных агентов под разные типы задач;
- у каждого агента свой отдельный контекст;
- у каждого агента свой отдельный workspace;
- у каждого агента своя отдельная память;
- можно добавлять новых агентов под отдельные направления и проекты без переделки ядра.

Система должна быть пригодна для ежедневной личной работы, разработки, продуктовой проработки, контента и проектной координации.

## 2. Базовый принцип системы

Система строится по модели:

`Алекс -> Orchestrator -> профильные агенты`

При этом:

- Telegram используется как основной интерфейс;
- Orchestrator является главной точкой входа;
- профильные агенты являются отдельными ботами и отдельными серверными агентами;
- каждый профильный агент изолирован по памяти, workspace и истории задач;
- добавление нового агента должно быть стандартной процедурой.

## 3. Основные сценарии использования

Система должна поддерживать 2 режима:

1. Работа через Orchestrator
- Алекс пишет одну задачу;
- Orchestrator определяет нужного агента;
- Orchestrator направляет задачу исполнителю;
- Orchestrator собирает и возвращает результат.

2. Работа напрямую с профильным агентом
- Алекс заходит в нужный чат конкретного агента;
- пишет задачу напрямую;
- агент работает внутри своего контура;
- при необходимости агент запрашивает помощь через Orchestrator.

## 4. Архитектурная модель

## 4.1. Общая схема

Система состоит из следующих уровней:

1. Telegram interface layer
- личный чат с Orchestrator;
- отдельные личные чаты с профильными агентами;
- при необходимости отдельные проектные чаты.

2. Orchestration layer
- маршрутизация задач;
- task registry;
- state management;
- retry и timeout policy;
- loop prevention;
- escalation logic.

3. Agent runtime layer
- отдельный runtime под каждого агента;
- отдельные роли;
- отдельные prompts;
- отдельные workspaces;
- отдельные outputs.

4. Memory / storage layer
- отдельная память каждого агента;
- общий реестр задач;
- архивы;
- notes по проектам;
- search / semantic retrieval.

## 4.2. Telegram bot-to-bot policy

Допускается использование Telegram Bot-to-Bot Communication для видимой маршрутизации.

Но:

- Telegram не является единственным источником истины;
- server state обязателен;
- Orchestrator остается центральной точкой маршрутизации;
- профильные агенты не должны образовывать свободную mesh-сеть.

## 5. Состав персональной команды

## 5.1. Основной состав MVP

### 1. Orchestrator

Назначение:

- главная точка входа;
- принимает все общие задачи;
- решает, кому отдать задачу;
- собирает ответ;
- контролирует межагентное взаимодействие.

Задачи:

- routing;
- decomposition при необходимости;
- status tracking;
- aggregation результатов;
- escalation.

### 2. Frontend Agent

Назначение:

- вся frontend-разработка;
- UI;
- интерфейсы;
- React / Next / frontend-архитектура;
- интеграция дизайна в код.

Задачи:

- верстка;
- компоненты;
- страницы;
- состояния интерфейса;
- frontend refactoring;
- client-side logic;
- визуальная реализация.

Требование:

- отдельный workspace;
- отдельная история задач;
- отдельная память по frontend-решениям и проектам.

### 3. Design Agent

Назначение:

- UI/UX дизайн;
- структура экранов;
- визуальные концепции;
- прототипирование;
- дизайн-система;
- референсы и креативное направление.

Задачи:

- проектирование экранов;
- дизайн-концепции;
- тексты для интерфейсов;
- moodboards;
- визуальные сценарии;
- handoff во frontend.

Требование:

- отдельный контекст от frontend;
- отдельная память по стилям, системам, референсам.

### 4. Backend Agent

Назначение:

- backend-разработка;
- API;
- база данных;
- интеграции;
- auth;
- очереди;
- серверная архитектура.

Задачи:

- endpoints;
- schema design;
- background jobs;
- integrations;
- storage;
- business logic.

Требование:

- отдельный workspace;
- отдельная память по серверной логике;
- отдельная проектная история.

### 5. Content Agent

Назначение:

- контент;
- тексты;
- презентации;
- сценарии;
- контент-планы;
- материалы для Telegram, сайтов, презентаций.

Задачи:

- посты;
- лендинговые тексты;
- продуктовые описания;
- презентации;
- брифы на визуал;
- контентная упаковка проектов.

Требование:

- отдельный контекст;
- отдельная память по тональности, проектам и контентным стратегиям.

### 6. Media Agent

Назначение:

- работа с фото, видео, визуальными материалами;
- креативные брифы;
- сценарии для съемок;
- структура медиа-контента;
- обложки, визуальные материалы, презентационные медиа.

Задачи:

- медиа-брифы;
- сценарии роликов;
- структура фото/видео-пакетов;
- постановка задачи на генерацию или продакшн;
- медиа-сопровождение контентных задач.

Требование:

- отдельный workspace;
- отдельный контекст от контентного копирайтинга.

### 7. Research Agent

Назначение:

- сбор и анализ информации;
- конкуренты;
- рынки;
- инструменты;
- технологии;
- сравнение решений;
- внешние сигналы.

Задачи:

- ресерч;
- фактчекинг;
- сравнительные обзоры;
- анализ референсов;
- поиск технических и продуктовых решений.

## 5.2. Расширяемый состав

Система должна позволять добавлять агентов:

- под отдельные проекты;
- под отдельные продуктовые линии;
- под отдельные исследовательские или операционные роли.

Примеры:

- `project_alpha_agent`
- `project_beta_agent`
- `qa_agent`
- `product_agent`
- `ops_agent`

## 6. Правило добавления агентов под отдельные проекты

Отдельный агент под проект создается только если:

- проект долгоживущий;
- у проекта свой отдельный контекст;
- у проекта свой собственный набор артефактов;
- задачи проекта мешают другим направлениям;
- нужен отдельный workspace;
- нужна отдельная история решений;
- нужно исключить смешение памяти с другими проектами.

Если проект маленький или временный, задача должна идти в уже существующего профильного агента.

## 7. Telegram-структура

## 7.1. Базовый интерфейс

Для персональной команды принимается следующая схема:

- один отдельный личный чат с `Orchestrator Bot`;
- один отдельный чат с `Frontend Bot`;
- один отдельный чат с `Design Bot`;
- один отдельный чат с `Backend Bot`;
- один отдельный чат с `Content Bot`;
- один отдельный чат с `Media Bot`;
- один отдельный чат с `Research Bot`.

## 7.2. Почему именно так

Причины:

- это личная система;
- нет нужды начинать с общей группы;
- отдельные чаты лучше изолируют контекст;
- проще работать как с “проектами” внутри Telegram;
- каждый агент выглядит как отдельный рабочий инструмент.

## 7.3. Допустимое расширение

Позже можно добавить:

- общую Telegram-группу-штаб;
- проектные группы;
- отдельные чаты на нескольких участников;
- командные топики, если появятся другие участники.

## 8. Правила оркестрации

Orchestrator обязан:

- принимать любые неструктурированные задачи;
- определять профильного исполнителя;
- создавать task record;
- передавать задачу агенту;
- контролировать timeout;
- избегать циклов;
- агрегировать ответы;
- возвращать понятный финальный результат.

Профильный агент обязан:

- работать только в своем домене;
- хранить память только своего домена;
- при необходимости запрашивать другой контур только через Orchestrator;
- не запускать произвольную bot-to-bot цепочку без контроля.

## 9. Изоляция контекста

Для каждого агента должны быть отдельными:

- Telegram bot identity;
- workspace path;
- memory namespace;
- outputs;
- scratch files;
- project logs;
- role files;
- skills policy.

Общими допускаются только:

- global task registry;
- agent registry;
- shared infrastructure configs;
- system-level logs.

## 10. Требования к разработке

Код системы должен содержать:

1. `agent_registry`
- хранение всех агентов;
- mapping bot -> runtime -> workspace;
- routing metadata.

2. `task_registry`
- task lifecycle;
- parent-child tasks;
- retries;
- timeout policy;
- dedupe state.

3. `telegram_adapter`
- receive updates;
- send messages;
- bot-to-bot rules;
- chat routing;
- loop protection hooks.

4. `orchestrator_engine`
- routing;
- decomposition;
- dispatch;
- aggregation;
- escalation.

5. `workspace_manager`
- создание workspace;
- изоляция путей;
- project folders;
- output folders.

6. `memory_layer`
- long-term memory;
- short-term working memory;
- searchable notes;
- project memories.

## 11. MVP-этапы

### Этап 1

- Orchestrator
- Frontend Agent
- Backend Agent

### Этап 2

- Design Agent
- Content Agent
- Research Agent

### Этап 3

- Media Agent
- агенты под отдельные проекты
- групповые и проектные Telegram-контуры

## 12. Что не входит в стартовый MVP

Не входит:

- свободная сеть ботов “все со всеми”;
- один общий workspace на всю команду;
- одна общая память на всех;
- автопилот без Orchestrator;
- автоматическое создание десятков проектных агентов без правил;
- полная автономность без контроля задач.

## 13. Критерий успеха

Система считается реализованной правильно, если:

- каждый ключевой агент изолирован;
- Orchestrator корректно маршрутизирует задачи;
- контекст frontend, design, backend, content и media не смешивается;
- можно напрямую работать с каждым агентом;
- можно добавлять новых проектных агентов без переделки основы;
- Telegram остается удобным интерфейсом, а не узким местом системы.
